# Доменная модель: календарь бронирования

Сервис по типу Calendly. В системе две условные роли — **владелец календаря** и **гость**.
Регистрации и авторизации нет: владелец — один заранее заданный профиль, гость бронирует
слот без создания аккаунта.

## Сущности

### Владелец (Owner)

Единственный, заранее заданный профиль. Используется по умолчанию в админской части.
Через API не создаётся и не изменяется.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | Идентификатор владельца |
| `name` | string | Имя владельца |
| `email` | string | Контактный email |
| `timeZone` | string | Часовой пояс владельца (IANA, напр. `Europe/Moscow`) |

### Тип события (EventType)

Шаблон встречи, который создаёт владелец. Гостю публично видны имя, описание и длительность.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | Идентификатор. **Задаётся владельцем** при создании |
| `name` | string | Название типа события (не пустое) |
| `description` | string | Описание |
| `durationMinutes` | int32 | Длительность встречи в минутах (> 0) |

### Слот (Slot)

Свободный интервал времени, на который можно записаться. Слоты формируются на 14 дней
от текущей даты; занятые интервалы из выдачи исключаются. Длина слота равна
`durationMinutes` соответствующего типа события.

| Поле | Тип | Описание |
|------|-----|----------|
| `start` | utcDateTime | Начало слота (ISO 8601, UTC) |
| `end` | utcDateTime | Конец слота, `end = start + durationMinutes` |

### Бронирование (Booking)

Запись гостя на конкретный слот выбранного типа события.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | Идентификатор брони. Генерируется сервером |
| `eventTypeId` | string | Тип события, на который сделана запись |
| `eventTypeName` | string | Название типа события (для списка встреч владельца) |
| `start` | utcDateTime | Начало встречи |
| `end` | utcDateTime | Конец встречи |
| `guestName` | string | Имя гостя (не пустое) |
| `guestEmail` | string | Email гостя |
| `note` | string? | Необязательная заметка от гостя |
| `createdAt` | utcDateTime | Момент создания брони |

## Сценарии

### Владелец

1. **Создаёт типы событий** — задаёт `id`, `name`, `description`, `durationMinutes`.
2. **Смотрит страницу предстоящих встреч** — единый список бронирований всех типов событий,
   отсортированный по времени начала.

### Гость (публичный сценарий, без аккаунта)

1. Смотрит **список типов событий** — доступны название, описание, длительность.
2. Выбирает тип события и открывает **календарь свободных слотов** на ближайшие 14 дней.
3. **Создаёт бронирование** на выбранный свободный слот, указывая имя и email.

## Правила

- **Глобальная занятость.** На одно и то же время нельзя создать две записи, даже если это
  разные типы событий. Время считается занятым, если интервал `[start; end)` пересекается
  с любым существующим бронированием.
- **Окно записи.** Доступные слоты формируются на 14 дней, начиная с текущей даты. Гость может
  записаться только на свободный слот из окна `[сегодня; сегодня + 14 дней]`.

## Принятые допущения

Фиксируем на уровне контракта, чтобы поведение было однозначным:

- Все моменты времени — `utcDateTime` (ISO 8601, UTC). Часовой пояс отображения — `Owner.timeZone`;
  часовые пояса гостей вне контракта на этом этапе.
- `id` типа события задаёт владелец, поэтому повторный `id` при создании — конфликт (`409`).
- Гость передаёт при брони **имя и email** (плюс опциональная заметка); аккаунт не создаётся.
- Некорректный `start` при брони (в прошлом, вне 14-дневного окна, не совпадает со свободным
  слотом) — ошибка `422`; уже занятое время — `409`; несуществующий тип — `404`.
- Отмена и редактирование брони, а также изменение/удаление типа события — вне рамок текущего
  контракта (набор эндпоинтов «по сценариям»).
