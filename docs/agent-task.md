# Задача агенту

Формулировка задачи для ИИ-агента, который готовит TypeSpec-спецификацию API-контракта.
Промпт самодостаточен: агенту не нужен доступ к другим материалам, кроме этого текста.

---

## Роль

Ты — инженер, проектирующий API-контракт. Твой результат — спецификация на **TypeSpec**
(`@typespec/http`), из которой при необходимости можно сгенерировать OpenAPI.

## Контекст

Нужно спроектировать сервис бронирования встреч по типу Calendly. В системе две условные
роли и **нет регистрации и авторизации**:

- **Владелец календаря** — один заранее заданный профиль, используется по умолчанию в
  админской части.
- **Гость** — бронирует слоты без создания аккаунта и без входа в систему.

## Доменные сущности

- **Владелец (Owner):** `id`, `name`, `email`, `timeZone`. Не создаётся через API.
- **Тип события (EventType):** `id` (задаёт владелец), `name`, `description`,
  `durationMinutes` (> 0).
- **Слот (Slot):** свободный интервал `start`/`end`; `end = start + durationMinutes`.
- **Бронирование (Booking):** `id` (сервер), `eventTypeId`, `eventTypeName`, `start`, `end`,
  `guestName`, `guestEmail`, `note?`, `createdAt`.

Все моменты времени — `utcDateTime` (ISO 8601, UTC).

## Сценарии

**Владелец:**
1. Создаёт типы событий (задаёт `id`, название, описание, длительность в минутах).
2. Просматривает страницу предстоящих встреч — единый список бронирований всех типов событий.

**Гость:**
1. Смотрит список типов событий (название, описание, длительность).
2. Выбирает тип события и получает свободные слоты на ближайшие 14 дней.
3. Создаёт бронирование на выбранный свободный слот, указывая имя и email.

## Правила и ограничения

- **Глобальная занятость:** на одно и то же время нельзя создать две записи, даже для разных
  типов событий (интервалы `[start; end)` не пересекаются).
- **Окно записи:** слоты формируются на 14 дней от текущей даты; запись возможна только в
  свободный слот из этого окна.
- `id` типа события задаёт владелец → повторный `id` при создании — конфликт.

## Что нужно сделать

Подготовь TypeSpec-спецификацию (`.tsp`), которая фиксирует контракт:

1. Опиши модели: `Owner`, `EventType`, `Slot`, `Booking`, тело запроса на создание брони,
   единый формат ошибки.
2. Опиши операции (эндпоинты) и разметь их тегами `owner` / `public`:

   | Метод | Маршрут | Роль | Ответы |
   |-------|---------|------|--------|
   | POST | `/event-types` | owner | 201, 409, 422 |
   | GET | `/event-types` | public | 200 |
   | GET | `/event-types/{eventTypeId}` | public | 200, 404 |
   | GET | `/event-types/{eventTypeId}/slots` | public | 200, 404 |
   | POST | `/bookings` | public | 201, 404, 409, 422 |
   | GET | `/bookings` | owner | 200 |
   | GET | `/bookings/{bookingId}` | public | 200, 404 |
   | GET | `/owner` | owner | 200 |

3. Для каждой операции задокументируй смысл и условия ошибок (`@doc`).
4. Разбей спецификацию на несколько файлов (модели отдельно от операций); точка входа —
   `main.tsp` с `@service`.

## Ожидаемый результат

- Набор `.tsp`-файлов: `main.tsp`, `models.tsp` и файлы операций по ресурсам.
- Валидация полей: непустые названия/имена, email-формат, `durationMinutes` > 0.
- Единый формат ошибок для 404 / 409 / 422.

## Критерии приёмки

- Каждый сценарий владельца и гостя отражён минимум одной операцией.
- Правило глобальной занятости выражено ответом `409` в создании брони.
- Окно 14 дней выражено выдачей слотов и ответом `422` при попытке брони вне окна.
- Публичные и владельческие операции различимы (теги/маршруты).
- Спецификация консистентна: типы и модели переиспользуются, ошибки единообразны.
