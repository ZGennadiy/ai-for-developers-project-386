# Дизайн: фронтенд календаря бронирования

Дата: 2026-07-09

## Контекст

Есть готовый API-контракт на TypeSpec (`spec/`) и доменная модель (`docs/domain.md`):
сервис бронирования встреч по типу Calendly с двумя ролями без авторизации —
**владелец** (единственный заранее заданный профиль) и **гость** (бронирует без
аккаунта). Бэкенда, реализующего контракт, в репозитории пока нет.

Задача этого шага — собрать рабочий UI, который реализует все сценарии контракта
и корректно работает с отдельно запущенным бэкендом (сейчас — с Prism-моком,
позже — с реальным сервером).

## Стек и инструменты

- **Vite + React + TypeScript** — сборка и dev-сервер.
- **shadcn/ui** (Tailwind + Radix) — UI-компоненты.
- **react-router-dom** — маршрутизация между гостевым и владельческим разделами.
- **TanStack Query** — запросы к API, кэш, состояния загрузки/ошибки, инвалидация
  после мутаций (бронь, создание типа события). Альтернатива — ручные
  `useEffect`/`useState` — отклонена: слишком много точек, где нужно вручную
  повторять загрузку/ошибки/лоадинги (слоты после брони, список встреч владельца
  после создания типа события).
- **react-hook-form + zod** — формы (бронь, создание типа события); интегрируется
  с `Form`-компонентом shadcn и даёт клиентскую валидацию, зеркалящую правила
  контракта (непустое имя, формат email, `durationMinutes > 0`), не заменяя
  серверную валидацию (404/409/422 остаются источником истины).
- Типы (`Owner`, `EventType`, `Slot`, `Booking`, `CreateBookingRequest`, `ApiError`)
  написаны вручную по `spec/models.tsp` — контракт компактный (4 модели),
  кодогенерация из OpenAPI для этого шага избыточна.
- Фронтенд — отдельная директория `frontend/` в корне репозитория (независимая
  часть приложения, согласно требованию шага).

## Маршруты и страницы

Общий `AppLayout`: хедер с названием сервиса и двумя ссылками навигации —
"Каталог" и "Кабинет владельца". Разделение ролей — навигационное, без проверки
доступа (авторизации в контракте нет).

### Гостевой раздел

| Маршрут | Назначение | Операции контракта |
|---|---|---|
| `/` | Список типов событий, карточки (название, описание, длительность) | `GET /event-types` |
| `/event-types/:eventTypeId` | Детали типа события + календарь свободных слотов на 14 дней, сгруппированных по дням | `GET /event-types/:id`, `GET /event-types/:id/slots` |
| (диалог на странице выше) | Форма брони выбранного слота: имя, email, заметка | `POST /bookings` |
| `/bookings/:bookingId` | Подтверждение брони (после создания — редирект сюда) | `GET /bookings/:id` |

### Раздел владельца

| Маршрут | Назначение | Операции контракта |
|---|---|---|
| `/owner` | Предстоящие встречи — единый список бронирований всех типов, сортировка по `start` (как отдаёт сервер) | `GET /bookings` |
| `/owner/event-types` | Список существующих типов событий + форма создания нового (`id`, `name`, `description`, `durationMinutes`) | `GET /event-types`, `POST /event-types` |

Профиль владельца (`GET /owner`) запрашивается один раз и показывается в хедере
раздела владельца (имя, email); `timeZone` используется для форматирования дат
(см. ниже).

## Отображение времени

Согласно допущению в `docs/domain.md`, часовой пояс отображения — `Owner.timeZone`,
часовые пояса гостей вне контракта на этом этапе. Поэтому все значения
`utcDateTime` (слоты, брони) форматируются в `Owner.timeZone` (полученном через
`GET /owner`), одинаково в гостевом и владельческом разделах — не в локальной
таймзоне браузера.

## Обработка ошибок и состояний

- Единый формат ошибки контракта — `ApiError { code, message }`.
- `404` (тип события / бронь не найдены) — состояние "не найдено" прямо на
  странице с сообщением и ссылкой на каталог.
- `409` при создании брони (слот уже занят) — тост с `message` от сервера +
  автоматический рефетч слотов (слот мог занять другой гость).
- `409` при создании типа события (`id` уже существует) — ошибка привязывается
  к полю `id` в форме.
- `422` (бронь вне 14-дневного окна / не совпадает со свободным слотом;
  невалидные поля типа события) — сообщение от сервера показывается у формы.
- Списки на время загрузки — skeleton-заглушки (shadcn `Skeleton`), не спиннер
  на весь экран.

## Разработка и запуск против бэкенда

Бэкенда в репозитории ещё нет. Для разработки используется **Prism** —
мок-сервер, поднятый из OpenAPI, сгенерированного из `spec/*.tsp`:

```bash
npx tsp compile spec/main.tsp --emit @typespec/openapi3
npx prism mock tsp-output/@typespec/openapi3/openapi.yaml
```

В `frontend/` добавляется npm-скрипт для запуска Prism и `.env` с
`VITE_API_BASE_URL` (по умолчанию адрес Prism, обычно `http://localhost:4010`),
чтобы позже переключиться на реальный бэкенд изменением одной переменной.

Ограничение мока: Prism проверяет форму запросов/ответов по контракту, но не
реализует бизнес-правила (реальная проверка занятости слотов, генерация окна
на 14 дней, персистентность между запросами) — это станет доступно только с
реальным бэкендом. Для целей этого шага (UI, соответствующий контракту) это
ожидаемо и достаточно.

## Вне рамок

- Аутентификация/авторизация — в контракте её нет.
- Редактирование/отмена брони, изменение/удаление типа события — вне рамок
  текущего API-контракта (см. `docs/domain.md`).
- Кодогенерация типов из OpenAPI — при расширении контракта в будущем можно
  пересмотреть.
