# E2E-сценарии

Сквозные сценарии поверх реального стека (Go-бэкенд + React-фронтенд), проверяемые
Playwright-тестами из [`frontend/e2e/`](../frontend/e2e). В отличие от Vitest+MSW-тестов
компонентов (сеть подменяется моками), здесь проверяется реальный HTTP-контракт между
фронтендом и бэкендом и реальные бизнес-правила (см. [`backend/internal/booking/store.go`](../backend/internal/booking/store.go)).

Бэкенд стартует без сид-данных типов событий — при запуске создаётся только `Owner`,
поэтому критический путь **обязательно** начинается с создания типа события владельцем:
иначе гостю нечего бронировать.

## Основной сценарий (автоматизирован: `booking-flow.spec.ts`)

| Шаг | Действие | Проверка |
|-----|----------|----------|
| 1 | Владелец на `/owner/event-types` создаёт тип события (уникальные `id`/`name` со случайным суффиксом) | Тост «Тип события создан»; новая запись видна в списке «Существующие типы» |
| 2 | Гость открывает каталог `/` | Карточка нового типа события видна (по названию) |
| 3 | Гость открывает `/event-types/:id` | Название/описание/длительность отображены; список слотов на 14 дней загружен |
| 4 | Гость выбирает первый доступный слот | Открывается диалог «Подтверждение записи» |
| 5 | Гость заполняет имя, email, заметку и жмёт «Подтвердить запись» | Переход на `/bookings/:bookingId` |
| 6 | Проверка страницы подтверждения | Название типа события, дата/время (совпадает со временем выбранной кнопки-слота), имя и email гостя, заметка |
| 7 (доп.) | Владелец открывает `/owner` | Новая встреча видна в таблице предстоящих встреч (по имени гостя и названию типа) |

## Вторичные сценарии

### 1. Глобальная занятость слота между разными типами событий

Ключевое бизнес-правило: бронь одного времени для любого типа события блокирует это же
время для всех остальных типов. Калькулятор слотов на фронтенде всегда запрашивает
актуальный список свободных слотов, поэтому уже забронированное время физически
невозможно выбрать через интерфейс повторно — кнопка для него просто не рендерится.
Из-за этого сценарий «пользователь кликает по занятому слоту и видит тост об ошибке»
воспроизводим только гонкой двух параллельных сессий и недетерминирован для e2e без
моков — эта ветка уже покрыта юнит-тестом с моками:
`frontend/src/components/bookings/BookingDialog.test.tsx` → `"shows a toast when the
slot is already taken (409)"`.

То, что стоит проверять сквозным тестом на реальном бэкенде — сам факт глобального
исключения интервала: владелец создаёт два разных типа события, гость бронирует первый
доступный слот для типа A, затем открывает тип B и убеждается, что тот же слот в списке
отсутствует (следующий доступный слот сдвинут на длительность события A).

Статус: необязательный отдельный файл `frontend/e2e/global-slot-conflict.spec.ts`,
не входит в обязательный минимум задачи.

### 2. Конфликт идентификатора типа события при создании (409)

Владелец пытается создать тип события с уже существующим `id`; ошибка привязывается к
полю «Идентификатор», значения формы не сбрасываются. Полностью покрыто юнит-тестом
`OwnerEventTypesPage.test.tsx` → `"attaches a 409 conflict to the id field"` с моками.
Статус: задокументировано, не автоматизировано в e2e — дублирование низкой ценности.

## Запуск

```bash
cd frontend
npm run test:e2e
```

Требует Go 1.25+ и Node 22+ в `PATH` — Playwright сам поднимает бэкенд (`go run .`,
порт 3000) и фронтенд (`vite`, порт 5173) через `webServer` в `playwright.config.ts`.

**Важно при локальном запуске:** вне CI Playwright переиспользует уже запущенный сервер
на нужном порту (`reuseExistingServer`), не проверяя его настройки. Если у вас уже
поднят `npm run dev` (например, по инструкции из `frontend/README.md`, указывающий на
Prism-мок на `:4010`) или `npm run mock`, тест молча пойдёт не в реальный бэкенд, а в
уже работающий процесс — и упадёт на непонятной проверке (сработано на практике: список
типов события показывал `string`/`1 мин` вместо реально созданных данных). Перед
`npm run test:e2e` остановите такие процессы (`lsof -nP -iTCP:5173 -sTCP:LISTEN` /
`-iTCP:4010` / `-iTCP:3000`, затем `kill PID`).
